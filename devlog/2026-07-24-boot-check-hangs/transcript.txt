● TL;DR: leaf_flags_in is innocent. The real bug is a kernel stack overflow that
  shreds the PML4 — the stage-0 boot path allocates the page-table root exactly
  one page below the 64 KB kernel stack, and the kernel overflows into it.
  Report written to docs/2026-07-24-stage0-stack-overflow-report.md (also sent
  to you directly).

  What I found

  Reproduction. bootstate_trace_check wedges 100%: serial shows the kernel going
  silent right after [serial] heap online, then — minutes later — a fresh
  [stage0] immutable selector banner. It's not a hang; it's an infinite
  triple-fault reboot loop, and the check scripts run QEMU with no subprocess
  timeout, so the loop looks like a hang forever.

  Evidence chain (live-debugged via qemu -s + lldb and qemu -d int):

  1. The fatal exception is an instruction-fetch #PF with CR2 == RIP on the
  kernel's own code, escalating #PF→#DF→triple fault because the exception
  handlers themselves became unfetchable. The register dump is damning: CR3 =
  0x0e01b000 (the PML4) lies between RSP = 0x0e018bb0 and RBP = 0x0e0204c8 — the
  call stack spans the page-table root.
  2. Stage-0 allocates the 64 KB stack, then PageTables::new(); OVMF allocates
  top-down, so the PML4 lands one page below the stack base, with no guard, and
  the identity map makes the whole area silently writable.
  3. I caught the corruption red-handed with a write watchpoint on PML4[511]: it
  fired from the kernel's own memmove (crt.rs:56) filling a stack buffer, with
  RSP already ~13.5 KB below the stack base. The debug build's deep
  boot-contracts sha256/generation frames (run during init_from_handoff →
  find_recovery_index) need ~78 KB of stack — the recent schema-binding commits
  pushed it over the 64 KB cliff.
  4. Why it symbolized to leaf_flags_in: the PML4's high entries get wiped
  first. A shallow trample leaves the handlers alive but puts stack garbage
  (e.g. a return address with bit 0 set) into PML4[384] — the ECAM scratch slot.
  leaf_flags_in dereferences entry & ADDR_MASK with zero validation, computes a
  ~4 PB "physical" address, and page-faults inside the walker — producing the
  [kernel fault] vec=14 rip∈leaf_flags_in + hlt_loop hang you saw. The fault RIP
  names the first reader of the corruption, not the writer.

  Ruled out: PMM recycling live frames (memory map verified in guest RAM — all
  table/kernel/stack frames are RESERVED), handoff ABI skew from 8a73ff1, and
  the huge-page mis-walk theory. Notably, check-transfer.py and the
  directory_check boot passed here (release builds have small frames and stayed
  within margin) — same unguarded geometry, just not over the cliff on this run.

  Recommended fixes (in the report, ordered by leverage)

  1. Stage-0: put an unmapped guard page below the kernel stack.
  2. Bump KERNEL_STACK_BYTES 64 KB → 256 KB.
  3. Harden the VMM walkers (leaf_flags_in etc.): reject PS-bit entries and
  bounds-check table addresses so corruption reports as a typed error instead of
  a fault inside the walker.
  4. Fix the scripts:
  check-bootstate-trace.py/check-rollback.py/check-release-trust.py boot the
  stale debug kernel even though the Justfile builds --release; also add
  timeout= to the QEMU subprocess.run calls so wedged boots fail loudly.

  One side effect of the investigation to know about: the repeated crash-boots
  consumed the boot attempts in /tmp/slime-os-bootstate-trace.img (it now boots
  boot-exhausted-known-good); the check script rebuilds it from scratch on each
  run, so nothing to clean up. All QEMU instances I started have been stopped.
